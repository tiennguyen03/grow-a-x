# Epic 4: Planet Interaction & Inspection

**Status:** 📋 Planned / Not Started  
**Primary owner:** Tien  
**Secondary coordination:** Nova for Matter Converter / economy integration  
**Recommended branch prefix:** `tien/planet-interaction-*`  
**Depends on:**
- Epic 0 traversal foundation: flight, boost, return-home, planet marker, star-system composition.
- Epic 1 Matter loop: dust collection, Matter HUD, server-side Matter profile.
- Epic 2 procedural planets: per-player Origin Worlds generated by `PlayerPlanetService`, orbiting the center star under `workspace.SpaceEnvironment.PlayerPlanets`.

---

## One-sentence goal

Give the player a clear, satisfying way to approach, inspect, understand, and eventually act on their own **Origin World** without mixing planet UX with Nova's economy internals.

---

## Product goal

The current game already has the main raw foundations:

1. The player can fly through space.
2. The player can collect Matter from dust motes.
3. Each player has a unique procedural Origin World.
4. The Origin World orbits the center star.
5. The player has a marker and return-home key to find the planet.

This epic adds the missing bridge:

```text
Fly through space
→ find Origin World
→ approach it
→ press E to inspect
→ camera frames the planet
→ player understands planet state
→ Matter Converter can later show planet interventions
```

The goal is not to build the full evolution system yet. The goal is to make the planet feel like a real interactable object and prepare a clean slot where Nova's Matter Converter / upgrade work can plug in.

---

## Design philosophy

### 1. The planet is the player's emotional anchor

The Origin World should feel like **my world**, not just scenery. The player should always know when they are near it and what they can do with it.

### 2. Inspection comes before upgrades

Before the player spends Matter, they need a place where spending Matter makes sense. The inspect view is that place.

Bad direction:

```text
Open random upgrade shop → buy +1 number
```

Good direction:

```text
Approach Origin World → inspect it → use Matter Converter → guide early life
```

### 3. Tien owns player-facing planet interaction

Tien's lane:
- planet approach UX
- inspect prompt
- inspect camera
- planet info panel
- UI/UX shell for planet context
- visual feedback hooks

Nova's lane:
- Matter spending
- server-side cost validation
- upgrade/intervention logic
- economy balancing
- Matter Converter internals

### 4. No fake overbuilding

This epic should not accidentally become civilization simulation. It should make the first planet interaction feel good and leave clear hooks for later systems.

### 5. Orbit-aware from the start

Player planets now orbit the center star. Therefore, every planet interaction system must target the **actual planet model's current pivot**, not a fixed `HOME_PLANET_POS`.

---

## Locked terms

Use these terms consistently:

| Term | Meaning |
|---|---|
| **Origin World** | Player-facing name for the player's first planet. |
| **HomePlanet** | Internal model/object naming used by planet systems. |
| **Matter** | Resource collected from dust motes. |
| **Matter Converter / Matter Core** | Nova's proposed menu/tool for upgrades and interventions. |
| **Intervention** | A Matter-powered action that affects planet development. |
| **Inspect Mode** | Focused camera/UI mode for looking at a planet. |

Avoid:
- generic upgrade shop language
- calling the player a colonist
- calling this a tycoon shop
- replacing Matter with Influence
- assuming planets are static

---

## Current system assumptions

Planet interaction should assume:

```text
workspace.SpaceEnvironment
└── PlayerPlanets
    ├── <player planet model>
    │   ├── PlanetBody
    │   ├── AtmosphereShell
    │   ├── SurfaceFeatures
    │   └── Clouds
```

Expected model attributes, depending on what Epic 2 already exposes:

```text
OwnerUserId
PlanetId
DisplayName
Seed
Archetype
Radius
WaterCoverage
CloudCoverage
AtmosphereThickness
RotationSeconds
```

The local player's planet should be found by:

```lua
planet:GetAttribute("OwnerUserId") == LocalPlayer.UserId
```

If any attribute is missing, the UI should fail gracefully and show fallback text.

---

## Out of scope for this epic

Do **not** build these in Epic 4:

- DataStore / saving
- complete evolution tier system
- server-side Matter spending logic
- Matter cost validation
- full Matter Converter internals
- multiplayer trade / war / diplomacy
- visiting other players' planets as a gameplay loop
- city lights / satellites / Dyson structures
- planet stat simulation
- random events / disasters
- prestige / Big Crunch / Big Bang reset
- server-authoritative movement
- large refactors of Matter or planet generation

# Sprint 4A: Planet Targeting + Approach Prompt

**Status:** Not Started  
**Owner:** Tien  
**Risk:** Low  
**Recommended branch:** `tien/4a-planet-approach-prompt`  
**Primary file:** `src/client/PlanetInteraction.client.luau`

## Goal

Detect the local player's own Origin World and show a simple prompt when the player is close enough.

This is the smallest possible planet interaction milestone.

## Player-facing behavior

When the player is far away:

```text
No prompt.
Planet marker still helps navigation.
```

When the player approaches their own planet:

```text
Press E to Inspect Origin World
```

When the player leaves range:

```text
Prompt disappears.
```

Pressing E can be logged or reserved, but full inspect mode comes in Sprint 4B.

## Implementation tasks

### 4A-01 Create `PlanetInteraction.client.luau`

Create a self-contained client LocalScript under:

```text
src/client/PlanetInteraction.client.luau
```

Responsibilities:
- wait for `Players.LocalPlayer`
- wait for character and `HumanoidRootPart`
- find `workspace.SpaceEnvironment.PlayerPlanets`
- find the planet model where `OwnerUserId == LocalPlayer.UserId`
- keep trying if the folder/model does not exist yet
- avoid errors if planet spawns late

### 4A-02 Local player planet lookup helper

Build helper functions:

```lua
local function getPlayerPlanetsFolder(): Folder?
local function findLocalPlayerPlanet(): Model?
local function getPlanetRadius(planet: Model): number
local function getPlanetDisplayName(planet: Model): string
```

Fallbacks:
- display name fallback: `"Origin World"`
- radius fallback: `50`

### 4A-03 Distance detection

Every frame or at a throttled interval, calculate:

```lua
local distance = (rootPart.Position - planet:GetPivot().Position).Magnitude
```

Use actual model pivot because the planet orbits.

Recommended constants:

```lua
PROMPT_DISTANCE = 170
PROMPT_HIDE_BUFFER = 20
UPDATE_INTERVAL = 0.1
```

Use a hide buffer so prompt does not flicker near the threshold:

```text
show if distance <= 170
hide if distance >= 190
```

### 4A-04 Prompt UI

Create a simple `ScreenGui`, not a 3D part prompt yet.

Suggested text:

```text
[E] Inspect Origin World
```

UI behavior:
- bottom-center or center-lower screen
- subtle dark panel
- white/cyan text
- hidden by default
- `ResetOnSpawn = false`
- do not duplicate on respawn

### 4A-05 Input detection stub

Detect E press with `UserInputService.InputBegan`.

In 4A, pressing E can do either:
- print a debug line in Studio, or
- set a local `wantsInspect = true` but not yet enter camera mode.

Keep this sprint focused on prompt visibility.

## Acceptance criteria

- Client finds only the local player's own planet.
- No errors if planet folder/model does not exist immediately.
- Prompt appears when close to own planet.
- Prompt hides when far away.
- Prompt follows the planet indirectly because distance uses current model pivot.
- Prompt does not appear for other players' planets.
- No duplicate prompt UI after respawn.
- No Matter/economy/server files touched.
- No Output errors.

4. Confirm prompt appears near the planet.
5. Fly away.
6. Confirm prompt disappears.
7. Press R to return near planet.

9. Reset character if possible.
10. Confirm no duplicate prompts.

# Sprint 4B: Inspect Camera Mode

**Status:** Not Started  
**Owner:** Tien  
**Risk:** Medium  
**Recommended branch:** `tien/4b-planet-inspect-camera`  
**Primary files:**

```text
src/client/PlanetInteraction.client.luau
src/client/SpaceMovement.client.luau   only if needed for movement lock
```

## Goal

Pressing E near the planet enters a focused inspect mode where the camera frames the orbiting planet and the player is temporarily prevented from flying away.

## Player-facing behavior

Near planet:

```text
[E] Inspect Origin World
```

Press E:

```text
Camera moves to a cinematic view of the planet.
Planet remains rotating/orbiting.
UI changes to inspect mode.
Player movement is paused.
```

Press E or Backspace:

```text
Camera returns to normal.
Player regains control.
```

## Implementation tasks

### 4B-01 Inspect state machine

Add state values:

```lua
local inspectMode = false
local currentPlanet: Model? = nil
```

Functions:

```lua
enterInspectMode(planet: Model)
exitInspectMode()
updateInspectCamera(dt)
```

State rules:
- cannot enter if no planet
- cannot enter if dead / no character root
- cannot enter if already inspecting
- exiting should always restore camera even if planet disappears

### 4B-02 Camera mode

On enter:

```lua
camera.CameraType = Enum.CameraType.Scriptable
```

On exit:

```lua
camera.CameraType = Enum.CameraType.Custom
camera.CameraSubject = humanoid
```

Track the planet every frame with `RunService.RenderStepped` because the planet orbits.

Camera position should be relative to the planet's current pivot:

```text
planetPos = planet:GetPivot().Position
starPos = WorldConfig.CENTERPIECE_STAR_POS if accessible, otherwise fallback
viewDirection = direction from star to planet or camera-friendly direction
cameraDistance = planetRadius * 3.0 to 4.0
cameraHeight = planetRadius * 0.6
cameraPos = planetPos + viewDirection * cameraDistance + Vector3.new(0, cameraHeight, 0)
camera looks at planetPos
```

If `WorldConfig` is hard to require on client, use a fallback camera offset first and improve later.

### 4B-03 Smooth camera transition

Initial version can snap. Better version should lerp.

Recommended:

```lua
CAMERA_SMOOTH = 8
```

Every frame:

```lua
camera.CFrame = camera.CFrame:Lerp(targetCFrame, 1 - math.exp(-CAMERA_SMOOTH * dt))
```

### 4B-04 Movement lock

Inspect mode needs to stop the player from flying.

Minimal approach:
- on enter, set `HumanoidRootPart.AssemblyLinearVelocity = Vector3.zero`
- repeatedly zero velocity while inspecting

Better approach:
- add a small movement lock check inside `SpaceMovement.client.luau`
- use a local player attribute:

```lua
LocalPlayer:SetAttribute("PlanetInspectLocked", true)
```

Then in `SpaceMovement.client.luau`, if locked:
- lerp velocity to zero
- skip movement input application
- keep orientation stable

This is a targeted change and acceptable in Sprint 4B because Tien owns traversal.

### 4B-05 Exit controls

Exit inspect mode on:
- E pressed again
- Backspace
- Escape optional, but avoid fighting Roblox menu behavior

### 4B-06 Failure recovery

If planet disappears while inspecting:
- exit inspect mode
- restore camera
- clear movement lock

If character respawns while inspecting:
- exit inspect mode
- reacquire root part
- no duplicated UI

## Acceptance criteria

- Press E near planet enters inspect mode.
- Camera frames current orbiting planet.
- Camera tracks planet as it moves.
- Player movement stops or is locked while inspecting.
- Press E or Backspace exits inspect mode.
- Camera always restores to normal.
- Movement always restores to normal.
- No snapping bugs that leave camera Scriptable forever.
- No Matter/economy/server files touched.
- No Output errors.

## Manual test

1. Fly near Origin World.
2. Press E.
3. Confirm camera frames planet.
4. Wait 30 seconds.
5. Confirm camera keeps tracking the orbiting planet.
6. Try WASD/Shift while inspecting.
7. Confirm player does not fly away.
8. Press E or Backspace.
9. Confirm normal flight returns.
10. Reset character while inspecting.
11. Confirm camera does not remain stuck.

# Sprint 4D: Matter Converter Integration Hook

**Status:** Not Started  
**Owner:** Tien + Nova coordination  
**Risk:** Medium-high because this touches the boundary with Nova's economy/UI work  
**Recommended branch:** `tien/4d-matter-converter-context` or coordinate into Nova's branch  
**Primary files depend on Nova's implementation**

## Goal

Let inspect mode provide planet context to Nova's Matter Converter / Matter Core UI without implementing the economy inside the planet interaction system.

This sprint is about interface boundaries, not costs or upgrades.

## Design decision

The Matter Converter is the player's universal menu/tool.

The Origin World inspect mode provides context:

```text
Current context: Origin World
Available category: Planet Interventions
```

Nova's system owns:
- buttons
- costs
- server validation
- Matter deduction
- intervention unlocks

Tien's system owns:
- entering planet context
- leaving planet context
- camera/inspection UX
- passing the current planet identity to UI

## Possible UX options

### Option A: Inspect panel has button

```text
[Open Matter Converter]
```

Clicking it opens Nova's Matter Converter in Origin World context.

### Option B: Matter Converter button always exists

When inspecting a planet, opening the converter automatically shows the `Origin World` tab/context.

### Option C: Inspect mode itself contains converter slot

Inspect panel includes an embedded placeholder area where Nova's intervention UI can mount.

Recommended first implementation: **Option A** because it is simple and low-coupling.

## Implementation tasks

### 4D-01 Define context payload

When inspect mode is active, the context should be expressible as:

```lua
local context = {
    type = "Planet",
    planetModel = planet,
    planetId = planet:GetAttribute("PlanetId"),
    ownerUserId = planet:GetAttribute("OwnerUserId"),
    displayName = planet:GetAttribute("DisplayName") or "Origin World",
}
```

Do not send Instances over remotes unless needed. Client-side UI can hold the model reference locally.

### 4D-02 Create client-side signal or API

Possible low-risk options:

1. BindableEvent under PlayerGui:

```text
PlayerGui.PlanetInspectContextChanged
```

2. Shared client module:

```text
src/client/PlanetInspectContext.luau
```

3. Player attributes:

```lua
LocalPlayer:SetAttribute("InspectingPlanetId", planetId)
LocalPlayer:SetAttribute("InspectingPlanetDisplayName", displayName)
```

Recommended: **client module or BindableEvent**. Player attributes are easy but less structured.

### 4D-03 Add placeholder button

In inspect panel:

```text
[Open Matter Converter]
```

For now, button can:
- fire the local context signal
- print debug output if Nova's UI is not ready
- show placeholder text: `Matter Converter integration pending`

### 4D-04 Do not implement spending

Do not:
- deduct Matter
- call server remotes for purchase
- modify PlayerManager
- add EvolutionTiers logic

### 4D-05 Coordinate with Nova

Before implementing this sprint, agree with Nova:

```text
What is the Matter Converter UI file called?
How should it receive planet context?
Should it open from inspect panel or from always-on HUD?
Who owns the Open Matter Converter button?
```

## Acceptance criteria

- Inspect mode can expose current planet context.
- Context includes planet id/name/owner.
- UI has a safe placeholder for opening Matter Converter.
- If Nova's Matter Converter is not ready, nothing errors.
- No Matter spending implemented here.
- No server economy files touched unless coordinated.
- Handoff clearly states what Nova should connect.

## Manual test

## Focus areas

- smooth camera framing
- prompt not flickering
- clean transitions
- compatibility with planet marker
- compatibility with return-home
- orbit tracking edge cases

## Implementation tasks

### 4F-01 Prompt polish

Improve prompt UX:
- fade in/out
- show distance maybe optional

Suggested behavior:

```text

```

### 4F-02 Camera framing polish

Tune camera:
- camera distance based on planet radius
- camera height based on radius
- camera direction based on star-to-planet vector
- avoid camera clipping inside planet/atmosphere

Formula idea:

```lua
local awayFromStar = (planetPos - starPos).Unit
local cameraPos = planetPos + awayFromStar * (radius * 3.5) + Vector3.new(0, radius * 0.7, 0)
```

### 4F-03 Orbit tracking stability

Inspect camera should track:

```lua
planet:GetPivot().Position
```

not cached position.

### 4F-04 Marker interaction

Coordinate with `PlanetMarker.client.luau`:
- marker should hide while inspecting

- marker should reappear after exit if far enough

This may require a local player attribute:

```lua
LocalPlayer:SetAttribute("InspectingPlanet", true)
```

### 4F-05 Return-home compatibility

After pressing R and arriving near planet:
- inspect prompt should appear if within range
- no UI stuck state
- if R is pressed while inspecting, either ignore or exit inspect first

Recommended behavior:

```text

```

But avoid overcomplicating if not needed.

## Acceptance criteria

- Prompt/marker/inspect UI do not overlap awkwardly.
- Camera frames planet beautifully.
- Camera follows orbit smoothly.
- Exiting inspect never leaves UI stuck.
- Return-home and marker still work.
